Proposal

My Project will be in the format of an interactive webpage and will follow the theme of illusion.

The webpage will consist of a grid of 9 circles, each by defualt containing a visual composition of solid lines and shapes (the plan is to stick with compositions of solid lines and shapes but this may change depending on how the animation process goes).

When a circle is hovered over, the pattern or composition will become animated and will display some sort of illusion as it is animated. This will be done using either an animated video (created in after effects) or through CSS and Javascript, or even a mix of the two, depending on the indivividual situation.

Process

In the majority of the circles, the svgs are coded by hand. This is what consumed the majority of my time, but it is also what allowed to me to morph them in transition for a much better hover transition as you see in the Multiples website.

While each circle had a unique process and each had its own frustrations, glitches, and "why isn't this working moments", the general prcess of defining svg points in html and manpulating them in css on hover remains the same. To demonstrate this process, one circle in particular will be used.

To begin, a single square was created in svg code. This square consists of four points, or pairs of x and y values, which not only determine its size but also its position. For example, (1,1) , (3,1) , (3,3) , (1,3). These coordinates were set in HTML.

The single square was duplicated, and the points were all translated so that this new square was positioned beside the original, and then repeated. Each square was given its own id so that the points of their coordinate could be changed in CSS on an individual basis when the parent circle is in a hover state.

Now back to the individual square. By default, the value of its coordinates are determined by what is set out in the html. The coordinates of the first square that I created had coordinates d="M4.5,4 4.5,6 2.5,6 2.5,4". In order for any SVG to morph into a different shape on hover, there must be the same number of coordinates (in this case there are four) and the element must be given a transition.

When the parent circle is in hover state, the CSS at that point dictates what the coordinates of that square will be. The work around to having the same number of coordinates (4) while making it a triangle was to make two coordinates the same value. The new coordinates for what was once a square is d: path("M5,4 4,6 4,6 3,4"); (as written in CSS).

Each square has a unique set of coordinates in the svg code which is embedded in the HTML. When the parent circle is in hover, each square's coordinates change on a case by case basis. To make the pattern consistent, everything was transformed by the same amount.

In the case of the multicolour cube, the process was more simple. It consisted of only three SVG components; a square and two polygons.

When the parent circle is in hover state, the coordinates simply translate and do not change shape or size. The method of transforming the coordinates is the same as the previous circle, but the transformations that take place are differnt and much less complex.

Organization and consistent classing and tagging methods were incredibly important, because there are hundreds of SVGs and paths in the HTML. Generally, an element that would be animated was labelled as by firstly the element type, then which parent circle it belonged to, and then was given a number to differentiate it from its sibling elements.

Reflection

After completing all nine circles, I was quite proud of what I had created. This was my first time working with inline SVG, css animations and transormations (in this depth), and keyframe animations.

Coming into this project, I knew that creating these animations would take effort but I had no idea what I was walking into. Each circle took up to half a day to fully create, since they were all so unique that they each introduced their own problems that had to be solved.

If I were to redo it or do it over, I would definetely focus on making the site compatible between browsers. I typically don't give that much thought because most HTML and CSS features are cross-compatible, but that is not the case with complex animations of SVGs.
